Routing rules:

- Execute the child skill named in the job packet first. Use this matrix only to resolve ambiguity or repair a bad packet.
- During ideation, always perform at least one target-domain, near-neighbor, and far-neighbor discovery attempt, then actively screen candidates before idea generation. Use broad metadata-only discovery for the first search pass (`depth=deep`, `searchMode=deep`, LLM-augmented planning, citation/related-work expansion, high query/candidate caps); run `pre_idea_discovery_config_lint.py`; write `ABSTRACT_SCREENING_AUDIT.json` so every merged candidate has an abstract-read row or an explicit no-abstract metadata fallback; then require PaperNexus import_workflow completion/authoritative sync for every actionable `import`/`supplement` graph-import row, source-limited exception records for exhausted no-fulltext rows, split-reading evidence for `material_view` rows, an innovation slot map, and a committed proposal graph session when the remote exposes `proposal_graph_session` before `EXPERIMENT_IDEA_POOL.json` is accepted. Treat target-domain methods as closest-prior pressure unless the novelty audit proves they are not present in the current field; top-tier main method ideas should cite near/far-neighbor mechanism transfer, external-domain transfer, or cross-lane recombination. If proposal graph support is unavailable or diagnosis-only, record the fallback boundary before using legacy `idea_catalyst`/`research_controller`. Explicit degraded speculative ideation requires user approval with claim limits.
- For every stage that can write `.autoreskill/user_view/`, maintain `user_view/innovation_story/` as prose-first story design. It must explain why the method is credible and novel through current-field pressure plus neighbor/cross-lane transfer; it must not replace evidence artifacts or collapse into a contribution list.
- For paper-code surveys, preserve `.autoreskill/survey/` as the machine-readable audit surface even when Obsidian notes are the user-facing deliverable. The required chain is raw candidates -> repository static evidence -> code mechanism map -> innovation migration matrix. Valid repositories support mechanism feasibility, not performance claims.
- Every PaperNexus discovery result must pass through screening before graph work: raw candidates -> `ABSTRACT_SCREENING_AUDIT.json` -> `PAPER_SELECTION_SCORECARD.json` -> `GRAPH_IMPORT_PLAN.json` -> `IMPORT_WORKFLOW_STATUS.json` from `import_workflow` queue/status/wait for all actionable `import`/`supplement` rows, plus explicit source-limited exceptions for exhausted no-fulltext rows -> PaperNexus material view or split-reading evidence for `material_view` rows. Do not use raw discovery rows or source-limited exception rows directly as novelty, method, baseline, limitation, or citation evidence.
- After ideation, prefer targeted discovery over broad reruns: closest-prior closure at `idea_gate`, selected-idea import/material closure at `experiment_plan`, terminal negative/regressed result diagnosis after `experiment`, negative/contradictory evidence at `analysis`, novelty/citation/baseline objections at `review_pressure`, and unresolved related-work/citation placeholders at `writing` or `submission_ready`.
- Treat `IDEA_DECISION_LEDGER.json` as the lifecycle authority and `EXPERIMENT_LEDGER.json` as the run authority. `IDEA_OUTCOME_SUMMARY.json` is the analysis bridge between them; it may summarize failure or promotion, but it must not replace either upstream ledger.
- Treat failed experiment routing as a first-class workflow route. `EXPERIMENT_LEDGER.json` failure rows with `failure_class` and `next_action` must produce either a bounded `repair_failed_experiment` job or a rollback/replan route; do not leave them as generic experiment incompletion. Fewer than two same-idea repairs may stay in `experiment`; two failed same-idea repairs require changing track, idea, or innovation before another launch.
- Treat migrated code ideas as first-class lifecycle entries. They must be selected, parked, killed, or repair-needed in `IDEA_DECISION_LEDGER.json`; a wiki note or valid source repo cannot bypass idea_gate.
- Use Codex thread heartbeats only for PaperNexus literature discovery, PaperNexus graph import/authoritative sync, and experiment runtime/resource waits. Local stage advancement, ready repair packets, lint failures, planning, review, writing, and generic queues must continue in the bounded loop or stop with an explicit blocker instead of scheduling a heartbeat.
- If a legacy project already has `EXPERIMENT_IDEA_POOL.json` without canonical `ideation/PRE_IDEA_EVIDENCE_GATE.json`, run `legacy_pre_idea_reconcile.py` and route repair to pre-idea evidence expansion instead of treating the stage as complete. A misplaced gate such as `orchestrator/PRE_IDEA_EVIDENCE_GATE.json` must be rebuilt or relocated only after validating the required lane/split-reading artifacts.
- Keep `code` and `experiment` under `Coder` unless a job packet explicitly asks Researcher to reconcile scientific intent.
- Do not let a child skill advance `goal_state.json`. After artifacts are written, mark the job complete and run another tick.
- If a linter fails, update the job as failed or retry with the exact blocker instead of manually advancing the stage.
